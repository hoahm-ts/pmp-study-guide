🌊 Waterfall / Predictive SDLC

🇺🇸 English

Waterfall follows sequential phases with formal gates between each. Every phase produces documented artifacts that become the baseline for the next. Change is controlled and formal.

Best for: Fixed-scope projects with clear requirements, regulatory-heavy environments, infrastructure/construction, large enterprise programs.

🇻🇳 Tiếng Việt

Waterfall theo các giai đoạn tuần tự với cổng kiểm soát chính thức giữa mỗi giai đoạn. Mỗi giai đoạn tạo ra tài liệu trở thành baseline cho giai đoạn tiếp theo.

Phù hợp với: Dự án phạm vi cố định, yêu cầu rõ ràng, môi trường tuân thủ cao, hạ tầng, chương trình doanh nghiệp lớn.

📈 Chart Interpretation Guide

📅 Gantt Chart — How to Read It
ElementsHorizontal bars = task duration. Diamond ◆ = milestone. Arrows = dependencies. Red/bold bars = critical path tasks (zero float).
AxesX-axis: calendar time (days/weeks/months). Y-axis: task list ordered by execution sequence.
✅ HealthyAll bars complete before the project deadline. Critical path has positive float (> 0 days). Milestones stable across plan revisions. Workload spread evenly across time periods.
⚠️ WarningFloat = 0 on critical path → any slip = immediate project delay. Tasks stacked in a single period → resource overload. Milestones shifting right week after week → schedule erosion. Long sequential critical-path chain → high coordination risk.
ActionFast-track: Run tasks in parallel → reduces time, increases risk and coordination cost. Crash: Add resources to critical-path tasks → reduces time, increases cost. Always present both options with trade-offs to the sponsor before acting.
💰 Budget S-Curve (Planned vs Actual Spend) — How to Read It
ElementsTwo cumulative lines: PV (Planned Value — budgeted spend per plan) and AC (Actual Cost — money actually spent). Both form an S-shape: slow start → steep execution peak → gradual closure tail.
AxesX-axis: project timeline. Y-axis: cumulative cost ($). The S-shape is expected and normal.
✅ HealthyAC line tracks close to PV. AC slightly below PV is acceptable — verify work is being delivered, not just deferred. S-curves are similar in shape.
⚠️ WarningAC above PV = cost overrun. Widening gap over time = overrun worsening. AC flat while PV rises = under-reporting or stalled work. AC rising steeply while EV stays flat = burning money without delivering value.
ActionIf AC > PV and CPI < 1.0: project EAC = BAC ÷ CPI and report to sponsor immediately. Identify which work packages are overspending. A CPI below 1.0 for 3+ consecutive periods rarely recovers.
🔥 Risk Heat Map (Probability × Impact Matrix) — How to Read It
ElementsA 5×5 grid (or 3×3). Each cell = a risk severity zone, color-coded green/yellow/red. Each bubble on the grid = a specific identified risk, plotted at its probability and impact scores.
AxesX-axis: Impact (1 = negligible → 5 = catastrophic). Y-axis: Probability (1 = rare → 5 = near-certain). Top-right corner = highest risk zone.
✅ HealthyMost risks in the green/yellow zone. Risks migrating toward green over time (mitigation working). No new red-zone risks without a mitigation plan. Total risk count stable or declining.
⚠️ WarningMultiple risks clustering in red zone. New risks appearing weekly without owners. Risks marked "Accepted" without a contingency plan. Same risk appearing on every report with no movement.
ActionEvery red-zone risk needs an owner, a mitigation action, and a deadline — not just a color label. Review weekly. Any risk moving toward red triggers escalation. "Accepted" risks still require a written contingency plan.
📊 EVM Dashboard (PV / EV / AC / CPI / SPI / EAC) — How to Read It

Earned Value Management is the PM's primary health monitor. Each metric tells a different story:

MetricFormulaWhat It Tells YouTarget
PV — Planned ValueBudget baselineHow much work should be done by today (per the approved plan)Fixed at baseline
EV — Earned Value% complete × BACBudgeted value of work actually completedTrack close to PV
AC — Actual CostActual spendTotal money spent to dateTrack close to EV
CPI — Cost Performance IndexEV ÷ ACCost efficiency: value per dollar spent. >1 = under budget; <1 = over budget≥ 1.0 ✅ / < 0.9 ⚠️
SPI — Schedule Performance IndexEV ÷ PVSchedule efficiency: delivery rate vs plan. >1 = ahead; <1 = behind≥ 1.0 ✅ / < 0.8 ⚠️
CV — Cost VarianceEV − ACDollar gap between earned and spent. Negative = over budget.> 0 ✅
SV — Schedule VarianceEV − PVDollar gap between earned and planned. Negative = behind schedule.> 0 ✅
EAC — Estimate At CompletionBAC ÷ CPIProjected total cost at current efficiency. Use for all forecasts once CPI is established.≤ BAC ✅
ETC — Estimate To CompleteEAC − ACMoney still needed to finish the projectDeclining over time ✅
VAC — Variance At CompletionBAC − EACProjected final over/under on budget. Negative = finishing over budget.> 0 ✅

CPI Recovery Rule: If CPI has been below 1.0 for three or more consecutive reporting periods, it will not recover to 1.0. A project with CPI = 0.75 would need to deliver all remaining work for 25 cents on the dollar to finish on budget — nearly impossible. Use EAC = BAC ÷ CPI as your honest forecast and report it to the sponsor. Never say "we'll recover" without a concrete, funded corrective action plan.

📉 CPI / SPI Trend Chart (over time) — How to Read It
ElementsTwo lines — CPI and SPI — plotted over the project timeline. Reference line at 1.0 = perfect performance. Some dashboards add an EAC trend line as a third line.
AxesX-axis: reporting periods (weekly or monthly). Y-axis: index value (typically 0.0–1.5).
✅ HealthyBoth lines stable at or above 1.0. Gradual improvement in CPI over time. SPI naturally improves toward 1.0 in the final project phase as remaining work must be completed.
⚠️ WarningLines declining for 2+ consecutive periods → systemic problem, not a one-off blip. CPI sustained below 0.9 → use EAC = BAC÷CPI in all future reports. SPI may recover near project end; CPI recovery almost never happens without additional funding.
ActionShow the trend (not just this week's snapshot) to the steering committee. A single bad period = investigate. Three bad periods = require a formal corrective action plan with owner, target date, and budget impact analysis.
📌
Waterfall — Key PM Rule

Every phase produces a baseline. Changes to the baseline require formal CCB approval. Never skip the gate review — it's the only mechanism that keeps scope, cost, and schedule under control. EVM is your early warning system during execution.

🏃 Scrum SDLC

🇺🇸 English

Scrum delivers in fixed-length iterations (Sprints, typically 2 weeks). The Product Owner defines what gets built; the team decides how. Transparency through artifacts and ceremonies is the core control mechanism.

Best for: Product development with evolving requirements, digital products, startups, cross-functional teams with good collaboration, when fast feedback loops matter.

🇻🇳 Tiếng Việt

Scrum deliver theo các vòng lặp cố định (Sprints, thường 2 tuần). Product Owner định nghĩa cái gì cần xây dựng; team quyết định cách thực hiện.

Phù hợp với: Phát triển sản phẩm với yêu cầu thay đổi, sản phẩm kỹ thuật số, startup, team cross-functional tốt, khi feedback loop nhanh quan trọng.

📈 Chart Interpretation Guide

🔻 Sprint Burndown Chart — How to Read It
ElementsIdeal line (perfect diagonal from [Day 1, total SP] to [Day 10, 0 SP]). Actual line (daily remaining story points). Optional scope line (tracks SP added or removed mid-sprint).
AxesX-axis: sprint working days (Day 1–10 for a 2-week sprint). Y-axis: remaining story points.
✅ HealthyActual line tracks near the ideal line. Slight lag early (stories need startup time) is normal. Line reaches 0 on Day 10. Small scope adjustments are acceptable with PO sign-off.
⚠️ Warning Line flat for 2+ days → stories blocked or too large to start completing
Line trending upward → scope added mid-sprint (Scrum Master must protect the sprint boundary)
Not converging to 0 by Day 7 → sprint goal at serious risk; action required
Large single drop then flat → one big story completed, rest still blocked
Perfect diagonal every sprint → team may be gaming estimates to match capacity exactly
ActionFlag at daily standup if actual line is not converging by Day 5. Work with PO to descope lowest-priority stories to protect the sprint goal. A delivered partial sprint goal beats a fully missed one. Never add scope to a struggling sprint.
📊 Velocity Chart — How to Read It
ElementsBar chart: one bar per completed sprint. Each bar height = story points completed that sprint. Some tools show committed vs. completed bars side by side.
AxesX-axis: sprint number (chronological). Y-axis: story points completed.
✅ HealthyBars stable with low variance (±15–20%). Slight upward trend = team maturing. Committed ≈ Completed. At least 5 sprints of data before using velocity for release forecasting.
⚠️ Warning High variance (large spikes and valleys) → personnel changes, constant interruptions, or unclear stories
Consistent decline over 3+ sprints → process breakdown or morale issue; investigate in retrospective
Spike followed by crash → overtime/crunch (unsustainable; leads to burnout)
Committed always much higher than Completed → chronic overcommitting; estimation needs fixing
Committed always much lower than Completed → sandbagging or lack of confidence in estimates
ActionUse the average of the last 3–5 sprints for sprint planning — never the highest sprint. Never use velocity to compare teams (story points are not standardized). Never pressure teams to increase velocity; it leads to inflated estimates and technical debt. Declining velocity for 3+ sprints requires a focused retrospective on systemic root causes.
📉 Release Burndown Chart — How to Read It
ElementsIdeal line from [Sprint 1, total backlog SP] to [target release sprint, 0 SP]. Actual remaining SP plotted per sprint. Scope line (if backlog is growing or shrinking).
AxesX-axis: sprint number. Y-axis: total remaining story points for the release.
✅ HealthyActual line tracks or is below the ideal line (delivering faster than planned). Scope line flat or declining. Team consistently delivering toward the target release sprint.
⚠️ Warning Actual line above ideal line → release date at risk
Scope line rising → backlog growing faster than team is delivering (scope creep)
Actual and ideal lines parallel but offset above → consistently behind and not recovering
Flat actual line for 2+ sprints → delivery is blocked
ActionIf projected completion > target release sprint: (1) defer lower-priority scope to the next release, (2) add team capacity, or (3) negotiate the release date with stakeholders. Always show the data — never hide a slipping release. Projection formula: Sprints remaining = Remaining SP ÷ Average velocity.
📋 Cumulative Flow Diagram (CFD) — Scrum Context
ElementsStacked area chart showing item counts across sprint board columns over the sprint duration. Each colored band = a workflow stage (To Do, In Progress, Review, Done).
AxesX-axis: time (sprint days). Y-axis: cumulative item count. Width of each band = WIP in that stage. Slope of the Done band = daily throughput rate.
✅ HealthyDone band slopes steadily upward. "In Progress" band stays narrow (low WIP). Bands flow smoothly from left to right.
⚠️ Warning "In Progress" band widening → too many stories started, none finishing (encourage swarming on stories)
Done band flat → no stories completing (stories too large, or team context-switching)
"To Do" band shrinking rapidly while Done is flat → stories starting but not finishing
ActionIf "In Progress" band is bulging, encourage the team to swarm on the most advanced story (closest to Done) rather than starting new ones. Stories should be completed one at a time before pulling new work.

Scrum — Key PM Rule

Velocity is a forecasting tool, not a performance metric. Never use velocity to compare teams or pressure individuals. The sprint burndown tells you if the sprint goal is at risk — the velocity chart tells you if the team's capacity is stable over time. Both are needed.

📋 Kanban SDLC

🇺🇸 English

Kanban is a continuous flow system — no sprints, no iterations. Work is pulled when capacity is available. Control comes from WIP limits, explicit policies, and flow metrics rather than planning ceremonies.

Best for: Operations/support teams, maintenance work, teams with highly variable request types, bug-fix queues, service desks, teams not ready for full Scrum ceremonies.

🇻🇳 Tiếng Việt

Kanban là hệ thống luồng liên tục — không có sprints. Công việc được kéo khi có dung lượng. Kiểm soát đến từ giới hạn WIP, chính sách rõ ràng và số liệu luồng.

Phù hợp với: Team vận hành/hỗ trợ, bảo trì, yêu cầu biến thiên cao, hàng đợi sửa lỗi, service desk.

📈 Chart Interpretation Deep Dive

The six Kanban metrics work as a system. Read them together, not in isolation.

🌊 CFD (Cumulative Flow Diagram) — How to Read It
ElementsStacked area chart. Each colored band = a workflow stage. Total height = total items in the system. Rightmost edge = items completed (Done band). The width of any band = WIP in that stage. The slope of the Done band = throughput rate.
AxesX-axis: time (days/weeks). Y-axis: cumulative item count. Steeper Done slope = faster delivery. Wider bands = more WIP in that stage.
✅ HealthyBands roughly parallel. Steady upward slope on all bands. Done band slopes consistently upward. Band widths stable (WIP under control). Total height stable or slowly growing.
⚠️ Warning Band bulging (widening) in one stage → WIP accumulating = bottleneck immediately downstream of that stage
Done band flat → no items completing; delivery has stalled
Total height growing rapidly → more items entering than leaving (WIP explosion)
Bands converging to a narrow strip → work drying up; backlog needs replenishment
Stepwise Done band → batch releases (items held and released together, not continuous flow)
ActionWhere the band bulges = where to investigate and reduce WIP. The CFD is the team's X-ray — read it daily, not weekly. A bulge in "In Review" means PR bottleneck; in "Testing" means QA bottleneck. Act on the constraint.
📊 Lead Time Distribution (Histogram) — How to Read It
ElementsHistogram (bar chart). Each bar = a time bucket (e.g., "1–3 days," "4–6 days"). Bar height = number of items completed in that time range. Reference lines at p50, p85, p95 percentiles. Lead time = from item creation to delivery (includes all wait time).
AxesX-axis: lead time buckets (days). Y-axis: count of work items completed in each bucket.
✅ HealthyBars clustered in a narrow, predictable range. Right-skewed curve is normal and expected. p85 is close to the main cluster. Most items fall within the SLA target. Distribution consistent across different time periods.
⚠️ Warning Long right tail → many outliers taking far longer than the norm; investigate the slow items
Bimodal distribution (two separate peaks) → two types of work mixed that need separate Classes of Service
Wide, flat histogram → highly unpredictable system; tighten policies and WIP limits
Items beyond the SLA line → confirmed SLA breaches; stakeholders already affected
ActionUse p85 (85th percentile) for SLA commitments — not the average. The average is skewed by outliers. Investigate items in the right tail: what type were they? What stage took longest? Use that insight to update policies or WIP limits.
⚡ Cycle Time Scatterplot — How to Read It
ElementsEach dot = one completed work item. Reference lines at p50 (median), p85, and p95 percentiles. Items above the p95 line are extreme outliers. Cycle time = from when work started to when it was delivered (excludes pre-work wait time).
AxesX-axis: completion date (showing when items finished over time). Y-axis: cycle time in days (how long that item took once started).
✅ HealthyDots randomly scattered below the p85 line. No visible upward trend over time. Consistent density of dots (stable throughput). Outliers are rare and do not cluster.
⚠️ Warning Dots trending upward over time → cycle time degrading; system is getting slower
Cluster of dots above p85 in a specific period → an event caused slowdown (deployment freeze, team absence, external dependency)
Vertical cluster on one date → batch delivery (items held and released together; not healthy flow)
Increasing dot density near p95+ → systemic problem accumulating
ActionInvestigate every outlier above p95. Most common root causes: blocked by external dependency, wrong Class of Service applied, rework required, single-person bottleneck. Add a "blocked reason" field in Jira to track patterns. The scatterplot tells you that something went wrong; the board policy tells you what to do about it.
📈 Throughput Chart — How to Read It
ElementsBar chart showing items completed per time period (week or month). Often includes a rolling 4-week average line overlaid on the bars.
AxesX-axis: time periods (weeks). Y-axis: number of items completed per period.
✅ HealthyStable bars with small variance. Rolling average line flat or gently rising. Consistent throughput regardless of which team members are present (process-dependent, not person-dependent).
⚠️ Warning Declining trend over 4+ weeks → system degrading (WIP too high, quality rework, team morale, or interrupt rate)
Single-week spike followed by a crash → batch release behavior; hidden WIP building up off-board
Near-zero weeks → blocked system, outage, or team event; mark and exclude from forecasting
High variance week-over-week → unstable demand intake or irregular work patterns
ActionUse the rolling 4-week average as input to Monte Carlo forecasting — never use the best week or worst week. If throughput is declining: check WIP levels, blocked item count, and interrupt rate. Stable throughput is more valuable than high throughput. Predictability beats speed.
⏳ Aging WIP Chart — How to Read It
ElementsEach dot = one item currently on the Kanban board, plotted at its current workflow stage and the number of days it has been in that stage. Reference lines at p50 and p85 based on historical cycle time data.
AxesX-axis: workflow stages (board columns from left to right). Y-axis: days the item has been in its current stage.
✅ HealthyMost dots below the p50 line. No dots above p85 in active stages. Items appear on the chart one day and move forward the next (regular flow). Few items persisting across multiple daily checks.
⚠️ Warning Items above p85 → already exceeding normal cycle time; likely to breach SLA soon
Items above p95 → SLA breach in progress unless unblocked immediately
Many items concentrated in one stage → WIP violation or stage bottleneck
Item aging in "In Review" for days → PR stuck; unblock or escalate
Items at p85+ in "Done" column → delivery delay; deployment pipeline problem
ActionCheck the Aging WIP chart every morning standup. Items above p85 get immediate attention — assign a rescuer, remove the blocker, or escalate. Use it in standup: "What's aging that we haven't discussed?" Never let items silently age past their SLA without action. This chart is your daily intervention tool.
🎲 Monte Carlo Forecast — How to Read It
ElementsA probability distribution showing forecast completion dates or item counts. Runs thousands of simulations using historical throughput data. Typical output: "X% probability of completing N items by [date]." Often shown as a date histogram or cumulative probability curve.
AxesX-axis: forecast completion dates. Y-axis: probability percentage (0–100%). The curve shows the cumulative probability of completing the target scope by each date.
✅ HealthyTight distribution (narrow cone between p50 and p85). Forecast aligns with stakeholder commitments. Based on at least 10–20 throughput data points for statistical reliability.
⚠️ Warning Wide cone (p50 and p85 dates far apart) → high throughput variability; low predictability
p50 date already past the committed delivery date → >50% chance of missing target
p85 date past committed date → 85% chance of missing target; escalate immediately
Based on fewer than 10 data points → statistically unreliable; label as "preliminary estimate"
ActionUse p85 for stakeholder commitments — never p50 (that is a coin flip). Use p50 for internal planning and team goals. If the cone is too wide: reduce scope variability, tighten WIP limits, or separate by Class of Service before running the simulation. Never give a single-point date — always give a range with a confidence level.

Kanban — Key PM Rule

Stop starting, start finishing. WIP limits exist for a reason — a team that violates them constantly is telling you something about their process, not their discipline. Use the CFD and Aging WIP chart as your daily early warning system, not as reports you generate after the fact.

🔀 Hybrid SDLC

🇺🇸 English

Hybrid combines predictive governance (milestones, formal approvals, change control) with agile delivery (sprints, backlog, iterative releases). Common in enterprises that need executive reporting but want team-level agility.

Best for: Regulated industries with fixed compliance gates, large programs with multiple delivery teams, projects with some fixed scope and some evolving features.

🇻🇳 Tiếng Việt

Hybrid kết hợp quản trị predictive (milestones, phê duyệt chính thức, change control) với delivery agile (sprints, backlog, release lặp lại). Phổ biến trong doanh nghiệp cần báo cáo cấp cao nhưng muốn agility ở cấp team.

📈 Chart Interpretation Guide

🚦 Milestone RAG Dashboard — How to Read It
ElementsA table or visual board listing all major milestones. Each row shows: milestone name, planned date, current forecast date, delta (days variance), owner, and a written narrative for any non-green status. RAG = 🟢 On Track / 🟡 At Risk / 🔴 Behind.
StructureRows ordered by date (next milestone at top). Columns: Status | Milestone | Planned Date | Forecast Date | Delta | Owner | Narrative.
✅ HealthyAll milestones green with forecast ≤ planned date. Amber milestones have active, named recovery plans. No milestone jumps from green directly to red without an amber phase (early warning is working).
⚠️ Warning Multiple amber milestones simultaneously → systemic delivery problem, not isolated events
Red milestone → formal escalation required; sponsor must be informed the same day
Amber with no owner or no action plan → cosmetic status tracking, not real management
Milestone dates shifting right over multiple reports → underlying problem not being addressed
ActionPM must provide a written narrative for every amber/red status — the color alone is not enough. Narrative must include: root cause, impact on downstream milestones, recovery plan, and whether a sponsor decision is needed. Present at every steering committee. "It's amber but we're watching it" is not a plan.
📉 Release Burndown (Phase Gate Connected) — How to Read It
ElementsSame structure as Scrum release burndown, but in a Hybrid context it connects directly to a formal phase gate milestone. The target date = gate date approved by the steering committee. Budget implications are visible.
AxesX-axis: sprint numbers leading up to the phase gate. Y-axis: remaining story points in the phase scope. Vertical marker at the gate sprint.
✅ HealthyActual line tracks the ideal line or is below it. Projected completion (SP remaining ÷ average velocity) = at or before the gate sprint. Scope stable. No rising scope line.
⚠️ Warning Projected completion sprint > gate sprint → trigger a formal change request immediately
Scope line rising while gate date is fixed → CCB must review which scope to defer
Velocity declining while gate approaches → double risk; escalate at next steering meeting
Gate sprint is 1–2 sprints away but significant SP remain → gate is at serious risk
ActionUnlike pure Scrum (where the team self-adjusts), Hybrid requires formal governance approval to change a phase gate date. When the burndown shows the gate is at risk: immediately raise a change request through the CCB to defer scope, extend the gate, or add resources. The PM bridges sprint velocity data to governance decision-making.
📊 EVM at Milestone Level (Hybrid) — How to Read It

In Hybrid, EVM is measured at phase gate level, not sprint level. Teams deliver via sprints inside each phase, but governance tracks cumulative EVM for the entire phase at each gate.

Metric at GateWhat It RepresentsThreshold
PV at gateTotal planned budget from project start to this phase gateFixed at baseline
EV at gate% of phase deliverables formally accepted × total phase budgetTrack close to PV
AC at gateTotal actual spend across all sprints in this phaseTrack close to EV
CPI at gatePhase cost efficiency (EV ÷ AC). Below 1.0 = phase overspent relative to value delivered≥ 1.0 ✅ / < 0.9 ⚠️
SPI at gatePhase schedule efficiency (EV ÷ PV). Below 1.0 = phase deliverables behind plan≥ 0.9 ✅ / < 0.8 ⚠️
EAC (full project)Projected total cost to completion (BAC ÷ CPI). Report this to sponsor at every gate.≤ BAC ✅

Gate Decision Rule: The steering committee uses CPI, SPI, and EAC to make a go/no-go decision. A gate review without EVM data is just a status report — it doesn't give the sponsor the information needed to protect the program budget. CPI < 0.9 and SPI < 0.8 together = high severity; sponsor must decide whether to proceed, re-baseline, or cancel.

🗺️ Feature-to-Milestone Heatmap — How to Read It
ElementsGrid where rows = features/epics and columns = milestones/phase gates. Each cell shows the RAG status of whether the feature will be ready by that gate. Used as a cross-reference between agile delivery and formal milestones.
StructureRows: Features (from the product backlog). Columns: Milestones (phase gates). Cells: 🟢 On track / 🟡 At risk / 🔴 Won't make it / ⚪ Not in scope for this gate.
✅ HealthyFeatures marked for a gate are green. Any amber feature has a clear action plan. Gate columns have a clear "must-have" set that is all green, and a "nice-to-have" set that can be amber.
⚠️ Warning Multiple must-have features in red for an upcoming gate → gate cannot proceed; escalate
Scope creep: cells being added to a gate that is already at risk
All features amber with no clear priority → PM has not made the "what must be done by this gate" decision
ActionPM must distinguish between must-have (gate-blocking) and nice-to-have (deferrable) features before each gate review. Share the heatmap with the PO and steering committee weekly in the final 4 sprints before a gate. Red must-have features trigger an immediate scope deferral or resource add discussion.

Hybrid — Key PM Rule

The PM's job in a hybrid model is to translate between layers — convert sprint velocity into milestone forecast language for executives, and convert compliance/regulatory requirements into backlog items the team can implement. Never let the two layers drift apart without a clear mapping.
